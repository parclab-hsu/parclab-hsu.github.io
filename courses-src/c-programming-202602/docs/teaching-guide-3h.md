# 3시간 강의 운영안

이 문서는 주차별 페이지를 실제 3시간 수업으로 운영하기 위한 강의자용 확장 자료다. 각 주차는 **도입 15~20분, 개념 설명 45~55분, 라이브 코딩 35~45분, 학생 실습 45~55분, 정리·형성평가 15~25분**을 기본 골격으로 한다.

!!! tip "운영 원칙"
    한 번에 많은 문법을 설명하기보다, 매주 하나의 문법이 **콘솔 → Arduino R4 → ROS2/로봇 사고방식**으로 어떻게 확장되는지 반복해서 보여 준다.

## 1주차 · OT·환경설정·왜 C인가

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~20분 | "왜 1학년이 C를 배워야 하는가"를 자동차 ECU, Arduino, ROS2 로봇 그림으로 연결한다. |
| 개념 | 20~60분 | C 언어의 위치, 컴파일 언어와 인터프리터 언어의 차이, 소스파일·목적파일·실행파일 흐름을 설명한다. |
| 환경 | 60~105분 | Visual Studio 2022 프로젝트 생성, `main.c` 추가, 첫 빌드와 실행을 함께 진행한다. |
| 보드 | 105~140분 | Arduino IDE에서 UNO R4 WiFi 보드 선택, 포트 확인, `Serial.println` 예제를 업로드한다. |
| Git | 140~165분 | GitHub 저장소 생성, 파일 추가, commit/push의 의미를 설명한다. |
| 정리 | 165~180분 | 설치 상태 체크, 컴파일 4단계 구두 확인, 다음 주 `printf` 예고. |

강의자는 첫 시간에 "환경설정은 수업의 일부"라는 점을 분명히 말한다. 설치 문제가 생긴 학생은 개별 문제로 두지 말고, 화면 공유로 공통 체크리스트를 따라가게 한다. `Hello, Mobility!`가 콘솔과 시리얼 모니터 양쪽에서 보이면 오늘 수업의 최소 목표는 달성한 것이다.

수업 중 질문:

- C 코드가 바로 CPU에서 실행되지 않고 컴파일을 거치는 이유는 무엇인가?
- Arduino에 업로드된 코드는 PC 프로그램과 무엇이 다른가?
- GitHub에 올린 코드는 왜 포트폴리오가 될 수 있는가?

## 2주차 · 표준 출력 printf

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 지난 주 `Hello` 코드를 다시 실행하고, 출력이 디버깅의 첫 도구임을 강조한다. |
| 개념 | 15~55분 | 형식 문자열, `%d`, `%f`, `%c`, `%s`, `\n`, `\t`를 자료형과 연결한다. |
| 판서 | 55~75분 | `printf("[%6.1f]", 3.14)`가 실제로 몇 칸을 차지하는지 칸 그림으로 보여 준다. |
| 라이브 코딩 | 75~115분 | 차량 대시보드 예제를 만들고 속도, RPM, 기어, 상태 문자열을 정렬 출력한다. |
| 실습 | 115~160분 | 학생별 "내 차 대시보드"를 만들고 최소 5개 항목을 표 형태로 출력한다. |
| 정리 | 160~180분 | 형식 지정자 짝 맞추기 문제, 폭·정밀도 예측 문제를 풀이한다. |

이 주차는 단순 출력이 아니라 **데이터 표현** 수업으로 다룬다. 센서값을 읽어도 사람이 볼 수 있게 출력하지 못하면 디버깅이 불가능하다. 숫자의 자릿수와 정렬은 로그를 읽는 품질을 결정한다.

확장 실습:

- 배터리 잔량을 `[%3d%%]` 형태로 출력한다.
- 온도를 `%.2f C`, 속도를 `%6.1f km/h`로 출력한다.
- 일부러 `%d`와 `%f`를 바꾸어 컴파일 경고와 이상 출력을 관찰한다.

## 3주차 · 입력·변수·자료형

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 출력만 하는 프로그램과 입력을 받아 반응하는 프로그램의 차이를 질문한다. |
| 개념 | 15~55분 | 변수, 자료형, 메모리 크기, `sizeof`, 정수와 실수의 표현 차이를 설명한다. |
| 입력 | 55~85분 | `scanf_s`의 `&` 의미를 "값을 넣을 주소를 알려 준다"로 설명한다. |
| 라이브 코딩 | 85~120분 | 거리와 시간을 입력받아 속도를 계산하고, 정수 나눗셈 실수를 일부러 만든다. |
| 실습 | 120~165분 | 학생이 직접 입력 기반 속도 계산기와 자료형 크기 출력 프로그램을 만든다. |
| 정리 | 165~180분 | `int`, `float`, `double`, `char` 선택 기준을 모빌리티 값 예시로 확인한다. |

이 주차에서 가장 중요한 메시지는 "자료형은 그릇의 크기이자 데이터 설계"라는 점이다. 학생이 단순히 `int`와 `double`을 외우는 데서 멈추지 않도록, 거리·시간·속도·상태·카운터를 각각 어떤 자료형으로 둘지 토론하게 한다.

판서 포인트:

- `int a = 7, b = 2; a / b`와 `(double)a / b`를 나란히 적는다.
- `scanf_s("%lf", &distance);`에서 `&`를 빨간색으로 표시하고, 12주차 포인터의 예고라고 말한다.
- `sizeof` 결과가 컴파일러와 플랫폼에 따라 달라질 수 있음을 짚는다.

## 4주차 · 연산자

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 속도 계산, 배터리 계산, 장애물 조건을 모두 연산자로 표현한다고 소개한다. |
| 개념 | 15~55분 | 산술, 관계, 논리, 대입, 증감 연산자를 짧은 코드로 비교한다. |
| 우선순위 | 55~80분 | 우선순위 표를 외우기보다 괄호로 의도를 드러내는 습관을 강조한다. |
| 비트 | 80~120분 | 센서 플래그를 1바이트 안에 담는 그림을 그리고 `|`, `&`, `~`, `^`를 설명한다. |
| 실습 | 120~165분 | 속도 계산기와 4방향 센서 플래그 관리 프로그램을 작성한다. |
| 정리 | 165~180분 | 비트 켜기/끄기/토글/확인 네 가지 패턴을 구두로 확인한다. |

비트 연산은 학생들이 어렵게 느끼므로, 숫자보다 스위치 그림으로 접근한다. `0000 0101`은 5라는 숫자이면서 동시에 0번 센서와 2번 센서가 켜진 상태라고 설명한다.

## 5주차 · 조건문 + LED 표정

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 장애물 거리별 STOP/SLOW/RUN 시나리오를 제시한다. |
| 개념 | 15~55분 | `if`, `else if`, `else`, 중첩 조건, 경계값을 설명한다. |
| switch | 55~80분 | 문자 명령을 `switch`로 처리하고 `break` 누락 실수를 보여 준다. |
| 라이브 코딩 | 80~120분 | 거리 배열을 순회하며 상태 문자열을 출력하는 콘솔 예제를 만든다. |
| Arduino | 120~160분 | UNO R4 LED Matrix 표정 예제를 업로드하고 조건에 따라 표정이 바뀌는 것을 확인한다. |
| 정리 | 160~180분 | 14.9, 15.0, 29.9, 30.0을 넣어 경계값을 토론한다. |

조건문은 안전 기준을 코드로 바꾸는 수업이다. "정지거리를 15cm로 둘 것인가, 20cm로 둘 것인가"처럼 실제 판단 기준을 먼저 정하고, 그 기준이 코드의 비교 연산자로 어떻게 표현되는지 연결한다.

확장 활동:

- `REVERSE` 상태를 추가해 10cm 미만이면 후진 경고 표정을 만든다.
- `switch`에서 `default`를 지우고 알 수 없는 명령을 넣어 본다.
- 상태 문자열을 대시보드 형식으로 출력한다.

| 구간 | 시간 | 내용 |

| 패턴 | 50~85분 | 합계, 평균, 최댓값, 최솟값을 누적 패턴으로 설명한다. |

반복문은 데이터 처리와 하드웨어 제어의 기본 리듬이다. 센서 샘플링, 주기적 제어, LED 애니메이션, 로봇 제어 루프는 모두 반복문으로 시작한다.

라이브 코딩은 일부러 `i <= n` 실수를 만들어 배열 범위 문제를 예고하면 좋다. 아직 배열을 깊게 배우기 전이지만, 반복 조건 하나가 프로그램 안정성에 큰 영향을 준다는 점을 보여 줄 수 있다.

## 7주차 · 함수 + 시리얼 제어

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 길어진 `main` 코드를 보여 주고 읽기 어려운 이유를 질문한다. |
| 개념 | 15~55분 | 함수 정의, 선언, 호출, 매개변수, 반환값을 설명한다. |
| 설계 | 55~85분 | `decide_state`, `show_state`, `run_command`처럼 역할별 함수 이름을 정한다. |
| 라이브 코딩 | 85~125분 | 주행 판단 코드를 함수로 분리하고 테스트 입력을 넣어 본다. |
| Arduino | 125~165분 | 시리얼 명령 `h/a/o/n/b`를 함수 호출로 연결한다. |
| 정리 | 165~180분 | call by value와 함수 프로토타입을 중간고사 범위로 정리한다. |

함수는 코드를 짧게 만드는 기술이 아니라 **책임을 나누는 기술**이다. 좋은 함수는 하나의 일을 하고, 이름만 보아도 목적을 알 수 있어야 한다. 이 원칙은 후반부 ROS2 노드에서도 그대로 적용된다.

- AI 사용은 해설 이후 복습용으로만 허용한다.
- 9주차 시작 전 `scope`, `lifetime` 단어를 미리 알려 준다.

| 구간 | 시간 | 내용 |
|------|------|------|

## 10주차 · 배열(1D) + 막대그래프

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 센서값 1개와 센서값 10개를 코드로 어떻게 저장할지 질문한다. |
| 개념 | 15~55분 | 배열 선언, 초기화, 인덱스, 길이, 범위 초과 위험을 설명한다. |
| 패턴 | 55~90분 | 배열과 반복문을 결합해 합계, 평균, 최댓값을 구한다. |
| 함수 전달 | 90~115분 | 배열을 함수에 전달할 때 길이를 함께 넘겨야 하는 이유를 설명한다. |
| 실습 | 115~165분 | 이동평균 필터와 LED 막대그래프를 작성한다. |
| 정리 | 165~180분 | `data[0]`, `data[n-1]`, `data[n]`의 차이를 확인한다. |

배열은 후반부 모든 데이터 처리의 시작점이다. 학생이 배열을 단순히 "여러 변수"로만 이해하지 않고, 메모리에 연속된 데이터이며 반복문과 함께 쓰인다는 점을 익히게 한다.

라이브 코딩에서는 `n = sizeof(data) / sizeof(data[0])` 패턴을 보여 주되, 함수 안에서는 이 방법이 작동하지 않는다는 점을 예고한다. 이 예고가 포인터 주차의 이해를 돕는다.

## 11주차 · 배열(2D)·통신 + WiFi

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | LED Matrix와 로봇 지도를 모두 격자로 볼 수 있음을 보여 준다. |
| 2D 배열 | 15~55분 | 행, 열, 중첩 반복, row-major 저장 방식을 설명한다. |
| 문자열 | 55~90분 | `char[]`, `'\0'`, `strcmp`, 문자열 버퍼 크기를 설명한다. |
| 통신 | 90~120분 | `"S,42.0,25.0,RUN"` 같은 텍스트 프로토콜을 소개한다. |
| WiFi 실습 | 120~165분 | UNO R4 WiFi 웹 서버 예제로 `/run`, `/slow`, `/stop` 명령을 처리한다. |
| 정리 | 165~180분 | 2D 배열, 문자열, HTTP 요청이 ROS2 토픽으로 이어지는 흐름을 요약한다. |

이 주차는 개념이 많으므로 "격자"와 "문자열 명령" 두 축으로 나누어 설명한다. 2차원 배열은 공간을 다루는 방법이고, 문자열은 프로그램 밖에서 들어오는 명령을 다루는 방법이다.

디버깅 포인트:

- WiFi 연결 실패는 코드보다 네트워크 환경 문제일 수 있다.
- 문자열 비교는 `==`가 아니라 `strcmp`를 사용한다.
- 버퍼 크기는 항상 널 문자 `'\0'`까지 고려한다.

## 12주차 · 포인터 기초

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | `scanf_s`에서 왜 `&`를 썼는지 다시 묻는다. |
| 개념 | 15~55분 | 주소, 포인터 변수, 역참조를 메모리 그림으로 설명한다. |
| 선언 읽기 | 55~80분 | `int *p`, `double *p`, `char *p`를 소리 내어 읽는 연습을 한다. |
| 배열 관계 | 80~115분 | 배열 이름과 시작 주소, 포인터 산술을 조심스럽게 소개한다. |
| 실습 | 115~165분 | 주소 출력, 값 변경, 포인터로 배열 순회를 작성한다. |
| 정리 | 165~180분 | `&x`, `p`, `*p`가 각각 무엇인지 그림으로 확인한다. |

포인터는 추상 설명만 하면 어렵다. 반드시 변수 상자, 주소 라벨, 화살표를 그려야 한다. `p`는 값이 아니라 주소를 들고 있고, `*p`는 그 주소로 가서 만나는 값이라는 점을 반복한다.

수업 중 금지하면 좋은 말은 "그냥 외우세요"다. 학생이 왜 주소가 필요한지 이해해야 13주차 call-by-reference와 15주차 배열 전달을 받아들일 수 있다.

## 13주차 · 포인터 활용

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 함수가 바깥 변수 값을 바꾸지 못했던 7주차 예제를 다시 꺼낸다. |
| 개념 | 15~50분 | 값 전달과 주소 전달의 차이를 `swap`으로 설명한다. |
| 다중 반환 | 50~85분 | 최댓값과 인덱스를 동시에 반환하는 예제로 포인터 인자를 사용한다. |
| 함수 포인터 | 85~115분 | 명령 문자와 실행 함수를 연결하는 맛보기 구조를 설명한다. |
| 실습 | 115~165분 | `swap`, 보정 함수, 최댓값+인덱스 반환 함수를 작성한다. |
| 정리 | 165~180분 | 포인터를 써야 하는 상황과 쓰지 않아도 되는 상황을 구분한다. |

이 주차에서는 포인터가 "위험한 문법"이 아니라 "함수가 원본 데이터와 협력하는 통로"라는 점을 강조한다. 원본을 바꿔야 할 때, 큰 배열을 복사하지 않고 처리할 때, 결과를 여러 개 돌려줘야 할 때 포인터가 필요하다.

실습 확장:

- 센서값 배열에서 최솟값과 위치를 동시에 구한다.
- `calibrate(double *v, double offset)`으로 거리값을 보정한다.
- 함수 포인터 배열로 `h/a/o/n` 명령을 처리한다.

## 14주차 · 구조체 + 연결리스트

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~15분 | 거리, 속도, 상태를 따로 관리할 때 생기는 문제를 질문한다. |
| 구조체 | 15~55분 | `struct`, `typedef`, 멤버 접근 `.`을 설명한다. |
| 포인터 | 55~80분 | 구조체 포인터와 `->`를 그림으로 설명한다. |
| 직렬화 | 80~115분 | 구조체를 `"S,42.0,25.0,RUN"` 문자열로 바꾸고 다시 파싱한다. |
| 실습 | 115~165분 | PC C 예제와 Arduino 구조체 패킷 예제를 실행한다. |
| 정리 | 165~180분 | 구조체와 ROS2 메시지의 공통점을 확인한다. |

구조체는 데이터 설계 수업으로 다루어야 한다. 변수 세 개를 묶는 문법을 넘어서, 어떤 값들이 함께 움직여야 하는지를 판단하는 능력이 중요하다. 센서 패킷, 로봇 자세, 학생 성적, 차량 상태는 모두 구조체의 좋은 사례다.

연결리스트는 맛보기 수준으로만 다룬다. 목표는 자료구조를 완전히 가르치는 것이 아니라, 구조체 안에 자기 자신을 가리키는 포인터가 들어갈 수 있다는 사실을 보여 주는 것이다.

## 15주차 · C↔ROS2·로봇·기말

| 구간 | 시간 | 내용 |
|------|------|------|
| 도입 | 0~20분 | 1~14주차 문법이 `/scan` 분석과 `/cmd_vel` 명령으로 연결되는 흐름을 보여 준다. |
| ROS2 개념 | 20~55분 | 노드, 토픽, publish/subscribe, `/scan`, `/cmd_vel`을 설명한다. |
| C 로직 | 55~90분 | `scan_logic.c`에서 배열 순회, 최솟값 탐색, 구조체 반환을 읽는다. |
| 브리지 | 90~120분 | `stella_n2_bridge.cpp`가 C 판단 결과를 ROS2 메시지로 바꾸는 과정을 설명한다. |
| 실습 | 120~160분 | 테스트 `/scan`을 발행하고 `/cmd_vel`을 echo로 확인한다. |
| 프로젝트 | 160~180분 | 기말 프로젝트 안전 규칙, 평가 루브릭, 발표 구조를 안내한다. |

마지막 주차의 핵심은 새 기술을 많이 소개하는 것이 아니라, 학생이 이미 배운 C 문법이 실제 로봇 소프트웨어 안에서 다시 등장한다는 사실을 확인하는 것이다. `/scan`의 `ranges[]`는 배열이고, 가장 가까운 장애물 탐색은 반복문이며, 정지/회피 판단은 조건문이고, 결과는 구조체로 묶인다.

안전 운영:

- 실제 로봇 연결 전에는 반드시 테스트 토픽으로 `/cmd_vel` 값만 확인한다.
- 바퀴를 띄운 상태에서 첫 실행을 한다.
- `stop_distance_m`은 크게, 속도는 작게 시작한다.
- 발표에서는 기능보다 "C 함수가 어떤 판단을 했는가"를 설명하게 한다.
