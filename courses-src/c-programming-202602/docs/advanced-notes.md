## 1. 정의되지 않은 동작(UB)과 시퀀스 포인트

C는 성능을 위해 많은 것을 **컴파일러 재량**으로 남긴다. 표준이 결과를 규정하지 않는 코드를
**정의되지 않은 동작(Undefined Behavior)** 이라 하고, 이런 코드는 **정답이 없다**(컴파일러·최적화에 따라 다름).

```c
int i = 1;
i = i++ + ++i;   // ❌ UB — 한 식에서 i를 여러 번 수정
a[i] = i++;      // ❌ UB — i 읽기와 수정 순서가 불명확
```
- **시퀀스 포인트**(`;`, `&&`, `||`, `?:`, 콤마, 함수 호출) 사이에서는 같은 변수를 **한 번만** 수정해야 한다.
- `printf("%d %d", f(), g())` 에서 `f`·`g`의 **호출 순서도 미정의**다(인자 평가 순서).

> 💡 **실전 시사점**: "이 코드의 출력은?" 문제에서 UB 표현이 보이면 **함정**이다. 잘 짠 문제는 UB를 쓰지 않는다.
> 코드를 짤 때도 한 줄에 부작용(`++`, 대입)을 몰지 말자.

## 2. 메모리 정렬(alignment)과 구조체 패딩

CPU는 특정 크기 경계에서 데이터를 읽을 때 가장 빠르다. 그래서 컴파일러는 구조체 멤버 사이에
보이지 않는 **패딩(padding)** 을 넣어 정렬을 맞춘다. **멤버 순서만 바꿔도 크기가 달라진다.**

```c
struct A { char c; int i; char d; };   // sizeof = 12  (1 + [3패딩] + 4 + 1 + [3패딩])
struct B { int i; char c; char d; };   // sizeof = 8   (4 + 1 + 1 + [2패딩])
```
- 실행 결과: `sizeof(A)=12`, `sizeof(B)=8` — **같은 멤버인데 크기가 다르다**.
- 큰 멤버부터 배치하면 패딩이 줄어 메모리가 절약된다(임베디드에서 중요).

> 💡 통신 패킷 구조체를 그대로 바이트로 보내면 패딩 때문에 깨질 수 있다 → 그래서 이 수업은 **문자열 직렬화**(`"S,42.0,..."`)를 쓴다.

## 3. 정수의 함정 — 오버플로와 부호 승격

정수는 유한 비트라 **범위를 넘으면 넘친다(overflow)**. 부호 있는 정수의 오버플로는 사실 UB지만,
2의 보수 환경에서는 보통 **음수로 순환**한다.

```c
int m = INT_MAX;        // 2147483647
printf("%d\n", m + 1);  // -2147483648  (최댓값 + 1 → 최솟값으로 순환)
```

더 은밀한 함정은 **부호 승격(usual arithmetic conversions)** 이다. `int`와 `unsigned`를 비교하면
`int`가 `unsigned`로 변환된다.
```c
unsigned u = 1;
int s = -1;
printf("%d\n", (s < u));   // 0 (거짓!) — -1이 unsigned로 바뀌면 4294967295 (큰 값)
```
- `-1 < 1` 은 참일 것 같지만, **`-1`이 거대한 unsigned로 승격**되어 거짓이 된다.

> 💡 배열 인덱스로 `size_t`(unsigned)를 쓸 때 `for (size_t i = n-1; i >= 0; i--)` 는 **무한 루프**(unsigned는 음수가 없음).

## 4. 부동소수점의 진실 — `0.1 + 0.2 != 0.3`

실수는 2진 분수로 저장되는데, `0.1`은 2진수로 **무한 반복소수**라 정확히 표현되지 않는다.

```c
double x = 0.1 + 0.2;
printf("%.17f\n", x);     // 0.30000000000000004
printf("%d\n", x == 0.3); // 0 (거짓!)
```
- 그래서 실수는 **`==`로 비교하면 안 된다**. 오차 허용치(epsilon)로 비교한다:
```c
if (fabs(x - 0.3) < 1e-9) { /* 같다고 본다 */ }
```
- 금액·좌표 누적처럼 오차가 쌓이는 계산은 특히 주의(자율주행 위치 추정에서도 실제 이슈).

## 5. 포인터의 위험 — 이 셋을 반드시 피하라

포인터는 강력한 만큼 위험하다. 크래시·보안 취약점의 대부분이 여기서 온다.

| 위험 | 무엇 | 예 |
|------|------|----|
| **댕글링 포인터** | 사라진 대상을 가리킴 | 지역변수 주소를 반환 |
| **메모리 누수** | `malloc` 후 `free` 안 함 | 반복될수록 메모리 고갈 |
| **이중 해제/미초기화** | 같은 걸 두 번 `free`, 초기화 안 한 포인터 역참조 | 크래시·UB |

```c
int* bad(void) {
    int x = 10;
    return &x;      // ❌ 함수 끝나면 x는 사라짐 → 반환된 주소는 무효(댕글링)
}
```
- 규칙: ① 포인터는 `NULL`이나 유효 주소로 **초기화**, ② `malloc` 하면 짝이 되는 `free`, ③ `free` 후 `NULL` 대입.

## 6. 타입 한정자 — `const` 와 `volatile`

- **`const`**: "이 경로로는 못 바꾼다"는 **약속**(컴파일러가 검사). 함수 매개변수에 `const`를 붙이면
  "이 함수는 원본을 안 바꾼다"는 계약이 된다 → `average(const double *arr, int n)`.
- **`volatile`**: "이 변수는 프로그램 밖(하드웨어·인터럽트)에서 바뀔 수 있으니 **최적화로 캐싱하지 마라**".
  임베디드의 **레지스터·센서 매핑 주소**에 필수. 없으면 컴파일러가 "안 바뀐다"고 오해해 잘못 최적화한다.
```c
volatile uint8_t *STATUS = (uint8_t*)0x40021000;  // 하드웨어 상태 레지스터
while (!(*STATUS & 0x01)) { }   // volatile 없으면 무한 루프로 최적화될 수 있음
```

## 7. 전처리기 매크로 함정

`#define`은 **함수가 아니라 글자 치환**이다. 괄호를 빼면 우선순위가 깨진다.

```c
#define SQ_BAD(x) x*x
#define SQ_OK(x)  ((x)*(x))
SQ_BAD(1+2)   // → 1+2*1+2 = 5   ❌
SQ_OK(1+2)    // → ((1+2)*(1+2)) = 9  ✅
```
- 매크로 인자와 전체를 **괄호로 감싸라**. 또 `SQ(a++)` 처럼 부작용 있는 인자는 **두 번 평가**되어 위험.
- 요즘은 타입 안전한 `static inline` 함수를 더 권장한다.

## 8. 시간 복잡도(Big-O) — 왜 알고리즘을 고르나

입력이 커질 때 연산량이 **어떻게 증가하는가**를 나타낸다. 상수는 무시하고 증가율만 본다.

| 표기 | 이름 | 예 | n=1,000,000 일 때 |
|------|------|----|-------------------|
| O(1) | 상수 | 배열 인덱스 접근 | 1 |
| O(log n) | 로그 | **이진 탐색** | 약 20 |
| O(n) | 선형 | 선형 탐색, 합계 | 1,000,000 |
| O(n log n) | 선형로그 | 좋은 정렬(병합·퀵) | 약 2천만 |
| O(n²) | 제곱 | **버블 정렬**, 이중 루프 | 1조 |

- `ex14_sort_search.c`에서 **정렬(O(n²)) 후 이진탐색(O(log n))** 을 쓰는 이유가 이것이다.
- 데이터가 커질수록 알고리즘 선택이 성능을 좌우한다 → 자율주행의 실시간 처리에서 핵심.

## 9. 함수 호출과 스택 프레임 — 재귀의 비용

함수를 호출하면 **스택 프레임**(지역변수·복귀주소·인자)이 스택에 쌓이고, 반환하면 걷힌다.
재귀는 이 프레임이 **깊이만큼 쌓이므로** 메모리를 쓴다.

```c
long fib(int n) {                 // 이 재귀는 O(2ⁿ) — 같은 값을 수없이 다시 계산
    if (n < 2) return n;
    return fib(n-1) + fib(n-2);
}
```
- `fib(50)`은 호출이 폭발해 사실상 멈춘다 → **반복문이나 메모이제이션**으로 바꿔야 한다.
- 너무 깊은 재귀는 **스택 오버플로**로 크래시. "재귀가 항상 우아한 답은 아니다."

> 💡 `ex10_recursion.c`의 `hanoi_moves`처럼 **선형 재귀**는 괜찮지만,
> 트리 재귀(fib)는 비용을 항상 따져라.

## 심화 자기점검
- [ ] `i = i++ + 1` 이 왜 UB인지 시퀀스 포인트로 설명할 수 있다
- [ ] 멤버 순서로 `sizeof(struct)`가 달라지는 이유(패딩)를 안다
- [ ] `-1 < 1U` 가 거짓인 이유(부호 승격)를 설명할 수 있다
- [ ] 실수를 `==`로 비교하면 안 되는 이유를 안다
- [ ] 댕글링 포인터·메모리 누수를 예로 든다
- [ ] `volatile`이 임베디드에서 왜 필요한지 안다
- [ ] `#define SQ(x) x*x` 의 문제와 해결(괄호)을 안다
- [ ] O(n²)과 O(n log n)의 실질 차이를 큰 n에서 설명한다
