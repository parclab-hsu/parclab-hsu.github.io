    웹 발표자료 제작¶

    이 장에서는 앞 단계에서 완성한 데이터 분석 결과를 링크로 나눌 수 있는 웹 슬라이드로 바꾸는 전 과정을 다룹니다. PPTX 파일이 아니라 GitHub Pages로 바로 공유할 수 있는 HTML 슬라이드를 만드는 것이 이 단계의 목표입니다.

    웹 발표자료 제작¶

    분석 결과를 링크로 나눌 수 있는 웹 슬라이드로 바꿉니다.

    여기서는 PPTX가 아니라 GitHub Pages로 공유할 수 있는 HTML 슬라이드라는 장점을 강조합니다. 링크 하나로 공유하고, 수정 후 재배포도 쉬우며, 메모·복사·차트 기능도 자유롭게 추가할 수 있습니다.

    분석 결과를 발표 흐름으로 바꾸기¶

    차트가 아니라 메시지가 슬라이드의 중심입니다.

    좋은 발표 슬라이드는 차트를 모아둔 것이 아니라 메시지가 흐르는 구조를 갖습니다. 각 슬라이드에는 다음 네 단계의 흐름이 담겨야 합니다.

    1. 결론 — 한 문장
    2. 근거 — 숫자/차트
    3. 해석 — 왜 중요한가
    4. 다음 질문 — 추가 확인

    이 흐름을 AI에게 맡길 때도 요청의 품질이 결과를 좌우합니다.

    • 부족한 요청 — "분석 결과로 발표자료 만들어줘." → 구조는 나오지만 메시지가 흐려지고 차트가 장식처럼 들어갈 가능성이 큽니다.
    • 더 나은 요청 — "청중은 대학 교수이고, 목표는 AI 에이전트 활용법 교육입니다. 데이터 분석 결과를 8장짜리 실습 발표자료로 구성해줘." → 청중, 목적, 장수, 말투, 차트 기준이 들어갑니다.

    제목은 주제가 아니라 결론으로 씁니다¶

    슬라이드 제목만 읽어도 메시지가 이어져야 합니다.

    슬라이드 제목을 '주제형'으로 쓰면 청중은 내용을 읽어야만 핵심을 파악할 수 있습니다. '결론형'으로 바꾸면 제목만 훑어도 발표의 흐름이 보입니다.

    주제형 제목 결론형 제목
    외국학생 현황 외국학생 총원은 3년간 증가했습니다
    중도탈락률 중도탈락률은 기준연도부터 확인합니다
    전형별 등록인원 전형별 등록인원 구성이 달라졌습니다
    성적 분포 성적 분포는 A/B 구간 변화가 핵심입니다

    AI에게 제목 수정을 요청할 때는 아래와 같이 조건을 명시하면 좋습니다.

    현재 슬라이드 제목을 모두 결론형으로 바꿔줘.
각 제목은 18자 안팎의 자연스러운 한국어 문장으로 만들고,
데이터가 말하지 않는 원인은 단정하지 마.

    슬라이드 재료를 네 칸으로 나눕니다¶

    AI에게 자료를 넘기기 전에 사람이 구조를 정합니다.

    HTML 슬라이드를 바로 만들어달라고 요청하기 전에, 각 슬라이드에 들어갈 재료를 아래 네 칸으로 정리해 두면 결과물의 품질이 크게 높아집니다.

    재료 예시 슬라이드에서의 역할
    메시지 외국학생 총원은 3년간 증가 제목 또는 핵심 문장
    숫자 380명 → 506명, +126명 근거 텍스트
    시각자료 추이선, 막대그래프, 누적막대 한눈에 보는 증거
    주의점 기준연도, 합계, 분모 확인 해석의 경계

    진행 팁 | 이 네 칸이 채워지지 않으면 HTML을 만들기 전에 분석으로 돌아갑니다.

    추천 발표 구성: 8장¶

    참가자가 자신의 분석 결과를 짧은 발표로 바꿀 때 쓰는 기본 틀입니다.

    아래 8장 구성은 AI 에이전트 활용 실습 워크숍에 맞춰 최적화된 기본 틀입니다. 각 장마다 핵심 메시지가 하나씩 들어가도록 설계되어 있습니다.

    장 제목 핵심 메시지 시각자료
    1 문제 제기 AI 에이전트로 업무 데이터를 다룰 수 있다 작업 흐름
    2 데이터 소개 4개 주제, 3개년 자료를 분석했다 데이터 지도
    3 품질 이슈 겉보기와 실제 데이터 구조가 달랐다 오류 도식
    4 인사이트 1 외국학생 총원이 증가했다 추이선
    5 인사이트 2 국가 구성 변화가 중요하다 막대그래프
    6 인사이트 3 전형과 성적 분포도 비교할 수 있다 요약 차트
    7 검증 합계와 기준연도를 확인했다 체크리스트
    8 적용 내 수업/학과 자료에 적용할 질문 다음 질문

    시간이 부족하면 3장만 만듭니다¶

    현장 실습에서는 완성형보다 작은 완성 단위가 안전합니다.

    워크숍 시간이 모자랄 때는 8장 전체를 억지로 만들려 하지 않아도 됩니다. 3장 MVP 구조로 핵심만 담아도 충분히 의미 있는 결과물이 나옵니다.

    • 1. 데이터 소개 — 어떤 파일을 읽었고 어떤 질문을 세웠는지 보여줍니다.
    • 2. 핵심 인사이트 — 차트 하나와 결론형 제목 하나로 가장 중요한 발견을 설명합니다.
    • 3. 검증과 다음 질문 — 합계, 기준연도, 해석상 주의점과 다음 확인 질문을 남깁니다.

    실습 팁은 다음과 같습니다.

    • 처음에는 8장을 억지로 만들기보다 3장을 제대로 만드는 편이 낫습니다.
    • 3장 구조가 안정되면 8장 구성으로 확장합니다.
    • Q&A 시간에는 내가 만든 3장 MVP를 가지고 피드백을 받습니다.

    구성안 요청 프롬프트¶

    먼저 HTML 파일부터 만들기보다 슬라이드 설계부터 잡습니다.

    HTML 파일을 바로 생성하도록 요청하기 전에 구성안(스토리보드)을 먼저 받아 검토하는 것이 훨씬 좋은 결과를 만듭니다. 아래 프롬프트를 그대로 사용해 보세요.

    지금까지 분석한 내용을 가지고
'AI 에이전트로 대학 행정 데이터를 분석하고 발표자료를 만드는 흐름'이라는 주제의
8장 발표자료 구성을 만들어줘.

슬라이드마다
1) 제목, 2) 핵심 메시지 한 문장, 3) 들어갈 차트나 표,
4) 발표할 때 말할 진행 멘트를 적어줘.
청중은 대학 교수이고, 실습형 워크숍에 맞는 자연스러운 말투로 작성해줘.

    구성안을 받은 뒤에는 아래와 같이 수정 지시를 추가할 수 있습니다.

    • 수정 지시 예 — "슬라이드 제목을 결론형으로 바꿔줘. / 차트가 없는 장에는 어떤 표가 필요한지 제안해줘. / 청중이 교수라는 점을 반영해 수업/교육행정 적용 예를 넣어줘."
    • 체크 포인트 — 장수가 맞는가? / 각 장마다 메시지가 하나인가? / 데이터와 무관한 일반론이 많지 않은가?

    구성안은 바로 통과시키지 않습니다¶

    AI가 만든 목차를 사람의 검수 질문으로 한 번 거릅니다.

    AI가 내놓은 첫 구성안은 대체로 그럴듯해 보이지만 느슨한 경우가 많습니다. 아래 네 가지 관점으로 한 번 더 점검하세요.

    • 한 장 한 메시지 — 결론은 하나인가? 두 메시지가 섞이지 않았는가?
    • 근거 연결 — 숫자나 표가 실제로 붙어 있는가?
    • 청중 적합성 — 교수법 맥락과 연결되었는가?
    • 해석 경계 — 데이터 범위를 넘어 원인을 단정하지는 않았는가?

    아래 검수 프롬프트를 활용하면 AI가 직접 약점을 찾아 수정안을 제시합니다.

    방금 만든 슬라이드 구성안을 검토해줘.
1) 한 슬라이드에 메시지가 두 개 이상인 장,
2) 근거 데이터가 부족한 장,
3) 청중에게 너무 일반적인 장,
4) 데이터 범위를 넘어 단정한 장을 찾아서 수정안을 제시해줘.

    웹 슬라이드 생성 프롬프트¶

    발표 화면, 차트, 메모, 배포 조건을 한 번에 지정합니다.

    구성안 검토가 끝났다면 이제 실제 HTML 파일을 생성할 차례입니다. 아래 프롬프트는 발표 화면 비율, 차트 방식, 키보드 이동, 발표자 메모, 배포 조건을 한꺼번에 지정합니다.

    위 구성안으로 GitHub Pages에 올릴 수 있는 웹 발표자료를 만들어줘.
조건은 다음과 같아.

- 발표자료/slide/index.html, styles.css, app.js로 구성
- 16:9 발표 화면
- 키보드로 이전/다음 이동
- 발표자 메모 토글
- 외국학생 총원 추이, 국가별 상위 국가, 중도탈락률, 성적분포 차트를 SVG로 렌더링
- 마지막에는 참가자가 다시 써볼 수 있는 프롬프트 카드 섹션 포함
- 외부 라이브러리 없이 GitHub Pages에서 바로 열리게 작성

    웹 슬라이드를 선택하는 이유와 실습 팁을 함께 기억해 두세요.

    • 왜 웹 슬라이드인가 — 링크로 공유 / 수정 후 재배포 쉬움 / 메모·복사·차트 기능 추가 가능
    • 실습 팁 — 완성형 HTML보다 구성안 먼저 / 1–2장 초안만 만들기

    파일 구조는 단순하게 둡니다¶

    초보자 실습에서는 복잡한 빌드 도구보다 바로 열리는 구조가 낫습니다.

    AI가 생성한 HTML 슬라이드는 세 개의 파일로 구성됩니다. 각 파일의 역할을 미리 알아두면 수정과 배포가 쉬워집니다.

    • index.html — 슬라이드가 들어가는 시작 파일입니다. GitHub Pages가 가장 먼저 여는 파일입니다.
    • styles.css — 글자 크기, 여백, 색상, 모바일 대응을 관리합니다.
    • app.js — 슬라이드 데이터, 이전/다음 이동, 발표자 메모, 차트 렌더링을 담당합니다.

    현장 기준 | React, 빌드, 패키지 설치 없이도 GitHub Pages에서 열리는 정적 HTML 구조로 시작합니다.

    GitHub Pages 배포 흐름¶

    정적 HTML은 빌드 없이 GitHub Pages에서 바로 열 수 있습니다.

    GitHub Pages 배포는 아래 다섯 단계로 진행됩니다.

    1. 1 계정 — GitHub 로그인
    2. 2 폴더 — 발표자료/slide
    3. 3 파일 — index.html, CSS, JS
    4. 4 커밋 — GitHub에 push
    5. 5 공유 — Pages 링크 확인

    배포 후에는 다음 두 가지를 확인하세요.

    • 예상 접속 경로 — https://<github-id>.github.io/workshop-hansung/
    • 공유할 때 주의 — GitHub 계정에 로그인되어 있어야 저장소에 push할 수 있습니다. 공유 전에는 실제 열리는 주소를 한 번 눌러보고 전달합니다.

    배포 후에는 주소를 직접 눌러봅니다¶

    push가 끝났다는 것과 발표자료가 열린다는 것은 다릅니다.

    배포가 완료된 뒤에는 반드시 세 단계로 화면을 직접 확인합니다.

    • 열림 확인 — Pages 주소를 새 브라우저 탭에서 직접 열어봅니다.
    • 캐시 확인 — 수정했는데 예전 화면이 보이면 CSS/JS 버전 또는 새로고침을 확인합니다.
    • 화면 확인 — 첫 장, GitHub 가입 장, Codex 안내 장, 차트 장, 마지막 장을 빠르게 넘겨봅니다.

    추가로 기억해 둘 체크리스트입니다.

    • 발표 전에 로컬 화면과 배포 화면이 같은지 비교합니다.
    • 모바일 공유까지 고려한다면 휴대폰에서도 첫 장과 핵심 장표를 확인합니다.
    • 링크 공유 전에는 비공개 자료나 실명 데이터가 들어가지 않았는지 다시 봅니다.

    좋은 웹 슬라이드로 고치는 기준¶

    AI가 만든 초안은 끝이 아니라 첫 편집본입니다.

    AI가 생성한 HTML 슬라이드를 그대로 쓰지 않고 아래 여섯 가지 기준으로 점검하고 편집하면 완성도가 높아집니다.

    • 제목 — 주제형보다 결론형으로 바꾼다. 예: 외국학생 현황 → 외국학생은 3년간 126명 증가
    • 차트 — 한 슬라이드에 메시지 하나만 남긴다. 축, 단위, 기준연도, 데이터 출처를 확인한다.
    • 화면 — 프로젝터에서 글자가 작지 않은가? 모바일에서는 핵심 내용이 먼저 보이는가?
    • 동작 — 이전/다음 이동이 되는가? 복사 버튼, 메모 토글이 발표를 방해하지 않는가?
    • 검증 — 합계가 맞는지, 기준이 섞이지 않았는지, 결론이 데이터 범위를 넘지 않는지 확인한다.
    • 청중 — 기능 나열보다 내 수업과 업무에 어떻게 적용할까로 연결한다.

    가독성은 실제 화면에서 봅니다¶

    CSS 숫자만 보고 판단하지 않고 프로젝터와 모바일 화면을 확인합니다.

    완성된 슬라이드는 반드시 실제 화면에서 가독성을 점검합니다. 네 가지 요소를 순서대로 확인하세요.

    • 글자 — 본문 최소 크기, 제목 줄바꿈, 버튼 안 글자 넘침을 봅니다.
    • 여백 — 카드가 너무 빽빽하거나 빈 공간이 과하게 크지 않은지 봅니다.
    • 차트 — 축, 단위, 범례가 읽히는지 확인합니다.
    • 모바일 — 작은 화면에서 핵심 내용이 먼저 보이는지 확인합니다.

    팁 | AI가 만든 디자인은 캡처해서 직접 보는 순간부터 진짜 편집이 시작됩니다.

    진행 메모까지 요청하기¶

    발표 준비와 실전 말하기를 한 번에 챙깁니다.

    발표자료를 완성한 뒤에는 각 슬라이드마다 진행 메모도 함께 요청해 두면 실제 발표 현장에서 큰 도움이 됩니다. 아래 프롬프트를 사용하면 실제로 말할 수 있는 자연스러운 문장을 얻을 수 있습니다.

    슬라이드마다 진행 메모를 작성해줘.
메모는 다음 형식으로 해줘.

- 말문 열기: 1문장
- 확인할 숫자: 1-2개
- 참가자에게 던질 질문: 1개
- 다음 슬라이드로 넘어갈 때 쓸 연결 문장: 1문장

너무 길게 쓰지 말고 실제 강의장에서 말할 수 있는 자연스러운 문장으로 써줘.

    진행 메모를 활용할 때 염두에 둘 점도 있습니다.

    • 진행 메모를 두면 좋은 이유 — 발표 흐름이 끊겨도 다음 문장으로 빠르게 회복할 수 있습니다.
